## Tension Resolutions

### Tension 1 — complexity_range: dict vs list

**Status: RESOLVED — requires spec edit AND code fix**

**Evidence gathered:**

- `AgentTriggers.complexity_range` is typed `Tuple[float, float]` (context.py line 55). This is the authoritative in-memory type.
- `_load_agent()` lines 228-236 handles only the YAML dict case via `.values()`. If a YAML list `[0.3, 1.0]` is passed, the `isinstance(..., dict)` guard evaluates to false and the fallback `(0.0, 1.0)` is used silently. This is a data-loss bug for any agent that uses the list form.
- The current 18 YAML agents use dict format `{min: 0.3, max: 1.0}`, which works with the existing loader.
- The design spec's Section 3.2 example uses dict format (`min: 0.3 / max: 1.0`) in the frontmatter reference. Section 3.4 (the normative senior-developer.md example) also uses dict format. Section Q7.1 recommends standardizing on list format. These three sections are mutually inconsistent.

**Decision:**

The new `.md` frontmatter format standardizes on YAML list format: `complexity_range: [0.3, 1.0]`. Rationale:

1. A YAML list maps directly to a Python tuple without the brittle `.values()` call, which is order-dependent on dict key order (Python dicts are insertion-ordered since 3.7, but this is a fragile assumption for a public configuration format).
2. The tuple annotation `Tuple[float, float]` is the canonical representation; YAML list is its most literal serialization.
3. The existing dict format in the 18 YAML files is a technical accident, not a deliberate design. The migration script must convert `{min: X, max: Y}` to `[X, Y]`.

**Required changes:**

1. **spec edit**: Section 3.2, under `triggers.complexity_range`, change example from:
   ```yaml
   complexity_range:
     min: 0.3
     max: 1.0
   ```
   to:
   ```yaml
   complexity_range: [0.3, 1.0]
   ```
   Add inline comment: `# YAML sequence; maps directly to AgentTriggers.complexity_range Tuple[float, float]`

2. **spec edit**: Section 3.4 (senior-developer.md example), same change in the frontmatter block.

3. **spec edit**: Section Q7.1 (Open Questions), update to reflect this resolution: "RESOLVED: list format standardized. See Tension 1 resolution in action-plan.md."

4. **code fix in registry.py**: The new `_build_agent_definition()` helper must handle both formats for backward compatibility:
   ```python
   raw = triggers_data.get("complexity_range", [0.0, 1.0])
   if isinstance(raw, dict):
       # Legacy dict format from existing .yaml files
       complexity_range = (float(raw.get("min", 0.0)), float(raw.get("max", 1.0)))
   elif isinstance(raw, (list, tuple)) and len(raw) == 2:
       complexity_range = (float(raw[0]), float(raw[1]))
   else:
       complexity_range = (0.0, 1.0)
   ```

**Loop-back required:** No. Decision is final. Senior-developer implements code fix; spec edits are straightforward substitutions.

### Tension 2 — Conditional IF: syntax deferral

**Status: DEFERRED — with explicit scope statement (requires spec edit)**

**Analysis:**

The `IF:` / `END IF:` syntax in Section 4.4 serves two distinct consumers: (a) LLMs reading prompt bodies as behavioral instructions, and (b) a future pipeline engine evaluating conditions programmatically to route execution. For consumer (a), the current free-form English condition syntax is fully adequate and correct — LLMs are excellent natural-language condition evaluators. For consumer (b), a formal machine-parseable grammar is required.

The pipeline engine does not yet exist (Stages 2–4 are unbuilt). Defining a formal grammar before the engine's evaluation semantics are understood would produce a grammar that fits the wrong abstraction. The correct sequencing is: build the engine, discover what condition types it needs to evaluate, then formalize only those condition patterns. Formalizing prematurely creates spec debt.

However, the current spec (Section 4.4) and the Q7.2 Open Question leave this ambiguous — neither explicitly documents the deferral or its scope boundary. This creates implementation risk: a developer reading the spec might attempt to implement condition parsing in the registry, which is wrong.

**Required changes:**

1. **spec edit**: In Section 4.4, after the "Condition syntax rules" block, add the following subsection:

   ```markdown
   #### Scope Boundary — Phase 1 vs Phase 2

   In Phase 1 (all work covered by this specification), `IF:` conditions are
   LLM-evaluated. The LLM reads the condition as a natural-language predicate
   against the named variables in its conversational context and decides whether
   to execute the enclosed CALL block. No parsing, no grammar validation, no
   engine evaluation.

   Machine-parseable condition evaluation — required for the pipeline engine to
   route execution without LLM involvement — is explicitly out of scope for this
   specification. It is Phase 2 work, gated on the pipeline engine design.
   The Phase 2 formal grammar should be a separate specification document.
   Do not implement condition parsing in registry.py or _load_md_agent().
   ```

2. **spec edit**: Close Q7.2 in Section 8: "RESOLVED AS DEFERRED. Conditions are LLM-evaluated in Phase 1. Phase 2 formal grammar is a separate specification. See Tension 2 resolution in action-plan.md."

**Loop-back required:** No for Phase 1. Phase 2 will require a new spec document.

### Tension 3 — Windows CRLF line endings in _load_md_agent()

**Status: REQUIRES_CODE_FIX**

**Analysis:**

The spec's `_load_md_agent()` implementation (Section 5.7) splits content on `"\n---\n"`. On Windows, files saved with CRLF line endings contain `"\r\n---\r\n"` as the frontmatter delimiter. The split will never match, causing every CRLF-encoded `.md` file to raise `AgentLoadError("no closing --- for frontmatter block")`.

Python's `open(..., encoding="utf-8")` in text mode on Windows does perform universal newlines translation by default (converting `\r\n` to `\n`) UNLESS `newline=""` is explicitly passed. However, relying on platform-specific default behavior is fragile and untestable cross-platform. The explicit fix is more robust.

Additionally, a UTF-8 BOM (`\xef\xbb\xbf`, which Windows Notepad and some editors emit) will prevent the `content.startswith("---")` check from passing. The fix for BOM is to use `encoding="utf-8-sig"` which strips the BOM automatically.

**Exact code fix for _load_md_agent():**

```python
async def _load_md_agent(self, md_file: Path) -> AgentDefinition:
    with open(md_file, "r", encoding="utf-8-sig") as f:  # utf-8-sig strips BOM
        content = f.read()

    # Normalize Windows CRLF to LF before any string operations
    content = content.replace('\r\n', '\n').replace('\r', '\n')

    if not content.startswith("---"):
        raise AgentLoadError(f"{md_file}: does not begin with YAML frontmatter (---)")

    # Split on the second occurrence of --- on its own line
    parts = content.split("\n---\n", 2)
    ...
```

**Required changes:**

1. **spec edit**: Section 5.7, `_load_md_agent()` code block — replace `open(md_file, "r", encoding="utf-8")` with `open(md_file, "r", encoding="utf-8-sig")` and add the two-line normalization after `f.read()`.

2. **spec edit**: Q1 in Section 8 (BOM question) — close it: "RESOLVED: Use utf-8-sig universally. This is the correct cross-platform choice for Windows-authored files. See Tension 3 resolution."

3. **unit tests**: Add `test_load_md_agent_crlf_line_endings` and `test_load_md_agent_bom_prefix` to the required test list in Section 6.2.

**Loop-back required:** No. This is a straightforward code correctness fix.

### Tension 4 — capabilities.py existence and vocabulary

**Status: PARTIALLY RESOLVED — spec language requires correction**

**Evidence gathered:**

`src/gaia/core/capabilities.py` EXISTS. It defines `AgentCapabilities` as a dataclass with fields: `supported_tools`, `supported_models`, `max_context_tokens`, `requires_workspace`, `requires_internet`, `requires_api_keys`, `supports_vision`, `supports_audio`, `supports_code_execution`. It has a `validate()` method that checks types and value ranges.

However, the file does NOT define a vocabulary list of capability string names. The spec claims: "Must use vocabulary defined in src/gaia/core/capabilities.py" (Section 3.2, capabilities field). This is incorrect — the file defines a dataclass for capability configuration, not a string registry for routing descriptor names like `"full-stack-development"` or `"api-design"`.

The two classes named `AgentCapabilities` in the codebase serve different purposes:
- `src/gaia/core/capabilities.py::AgentCapabilities` — runtime capability configuration dataclass (tools list, model list, boolean flags)
- `src/gaia/agents/base/context.py::AgentCapabilities` — routing/identity dataclass (capability strings list, tools list, execution targets dict)

These are distinct types. The spec conflates them.

**Required changes:**

1. **spec edit**: Section 3.2, capabilities field comment — change "Must use vocabulary defined in src/gaia/core/capabilities.py" to: "Should use a controlled vocabulary for pipeline routing. A formal vocabulary constant list (VALID_CAPABILITY_STRINGS) must be added to src/gaia/core/capabilities.py as Phase 2 work. Until then, use values consistent with existing agent YAML files."

2. **spec edit**: Section 6.1 Step 4 — change "Standardize capabilities: values against src/gaia/core/capabilities.py vocabulary" to "Standardize capabilities: values against the vocabulary being developed for src/gaia/core/capabilities.py (see Phase 2 VALID_CAPABILITY_STRINGS task)."

3. **spec edit**: Close Q7.4 in Section 8: "PARTIALLY RESOLVED: File exists. However it does not contain a string vocabulary list. VALID_CAPABILITY_STRINGS must be added to capabilities.py as Phase 2 work. The validation step in _build_agent_definition() should be skipped until that list exists."

4. **new Phase 2 task**: Add `VALID_CAPABILITY_STRINGS` constant to `capabilities.py` containing the canonical routing descriptor vocabulary derived from the 18 existing YAML agent files.

**Loop-back required:** No for Phase 1. Phase 2 implementation must add the vocabulary list before capability validation can be enforced at load time.

### Pseudocode

```
async def _load_md_agent(md_file: Path) -> AgentDefinition:

    1. Open file with encoding="utf-8-sig"  [handles BOM]
    2. Read full content as string
    3. Normalize: content = content.replace('\r\n', '\n').replace('\r', '\n')

    4. Guard: if not content.startswith("---"):
           raise AgentLoadError(md_file, "no frontmatter")

    5. Split: parts = content.split("\n---\n", maxsplit=2)
       Note: split("X", 2) on "---\nFRONTMATTER\n---\nBODY" produces:
         parts[0] = "---"         (before first separator)
         parts[1] = "FRONTMATTER" (between separators)
         parts[2] = "BODY"        (after second separator)
       But our content starts with "---\n", so:
         parts[0] = ""  or "---" depending on exact start
       IMPORTANT: The split must handle the opening "---" correctly.

    CORRECT SPLIT LOGIC:
    - Strip the leading "---\n" from content before splitting:
        if content.startswith("---\n"):
            content = content[4:]  # remove opening ---\n
        first_close = content.find("\n---\n")
        if first_close == -1:
            raise AgentLoadError(md_file, "no closing ---")
        frontmatter_text = content[:first_close]
        system_prompt = content[first_close + 5:].strip()  # skip "\n---\n"

    6. Parse: data = yaml.safe_load(frontmatter_text)
       Guard: if not data or not isinstance(data, dict):
           raise AgentLoadError(md_file, "empty or invalid frontmatter")

    7. Call: return self._build_agent_definition(
           data=data,
           system_prompt_override=system_prompt
       )
```

### _build_agent_definition() Key Implementation Notes

    1. agent_data = data.get("agent", data)  [handles both nested and flat YAML]

    2. triggers_data = agent_data.get("triggers", {})

    3. complexity_range parsing (handles both legacy dict and new list format):
       raw = triggers_data.get("complexity_range", [0.0, 1.0])
       if isinstance(raw, dict):

## Core Principles

- **Read before writing.** Always inspect existing code before modifying it. Understand
  the pattern before extending it.
- **Minimal change surface.** Implement exactly what was specified. Resist the urge to
  refactor adjacent code that is not part of the task.
- **Test-first thinking.** For every function written, identify the test case before
  writing implementation code. Tests are not optional.
- **Respect constraints.** Never exceed max_file_changes (20) or max_lines_per_file
  (500) per execution. If the task requires more, flag it and split the work.
- **No silent failures.** If a requirement is ambiguous, unclear, or contradicts existing

## Risk Register

### Risk 1 — Split logic produces incorrect frontmatter/body split for edge-case file structures

**Probability:** Medium. Files ending without a trailing newline, files with `---` in the first line of the body, or files with more than two `---` delimiters are all edge cases.

**Impact:** High. A wrong split produces either garbled YAML (parse error) or truncated system prompt (silent data loss).

**Mitigation:** The pseudocode in Deliverable 3 uses `content.find("\n---\n")` with an explicit strip of the opening `"---\n"` prefix, which correctly handles all standard cases. Unit tests `test_load_md_agent_body_preserves_special_characters` must include a body that contains `---` as a horizontal rule. Add a test with a body that starts immediately after the closing delimiter with no blank line.

---

### Risk 2 — dict.values() ordering in legacy complexity_range parsing

**Probability:** Low. Python 3.7+ guarantees dict insertion order. YAML dict parsing by PyYAML preserves source order. `{min: 0.3, max: 1.0}` will always yield `(0.3, 1.0)` in the correct order.

**Impact:** High if it fails. An agent with `min: 0.3, max: 1.0` loading as `(1.0, 0.3)` would never match any task (complexity score is always less than the "min" which is stored as max).

**Mitigation:** The new `_build_agent_definition()` helper uses explicit key lookup (`raw.get("min", 0.0)`, `raw.get("max", 1.0)`) instead of `.values()`, eliminating the ordering dependency entirely. This fix also applies retroactively to the existing YAML loader via code refactor.

---

### Risk 3 — Capability string vocabulary drift before VALID_CAPABILITY_STRINGS is implemented

**Probability:** High. Without enforcement, each agent author invents capability strings independently. After 18 migrations, the vocabulary will have inconsistencies (e.g., `"full-stack-development"` vs `"fullstack"` vs `"full_stack_dev"`).

**Impact:** Medium. Routing index built from inconsistent strings produces false negatives in `select_agent()` — the agent exists but is never selected because the capability query doesn't match.

**Mitigation:** Before migration, extract the complete set of capability strings from all 18 YAML files and normalize them. Document this canonical list in a comment block in `capabilities.py` even before `VALID_CAPABILITY_STRINGS` is formalized as a constant. Apply the normalized list consistently during migration.

---

### Risk 4 — Agent ID collision when both .yaml and .md files exist for the same agent

**Probability:** Medium during the migration period. If `senior-developer.yaml` and `senior-developer.md` both exist in `config/agents/`, `_load_all_agents()` will attempt to load both and the second load will silently overwrite the first (same `agent.id` key in `self._agents` dict).

**Impact:** Medium. Depending on glob ordering, either the old YAML or the new MD version wins. This is non-deterministic across platforms.

**Mitigation:** Add a collision check to `_load_all_agents()`: if `agent.id` already exists in `self._agents` and the new file has a different suffix, log a warning and skip the second file. Alternatively, load YAML files first and skip `.md` files whose ID is already registered. Document this behavior in the code and in the migration guide.

---

### Risk 5 — Template library staleness as spec evolves

**Probability:** High over time. The spec and templates are maintained separately; a spec change that adds a required frontmatter field will not automatically update the templates.

**Impact:** Medium. Ecosystem Builder generates agent files from stale templates, producing files missing the new required field, which then fail to load.

**Mitigation:** Section 9.5 (new spec addition) documents the maintenance obligation: template updates are required in the same commit as spec changes. Add a `templates/README.md` note: "These templates correspond to spec version X.Y.Z. When the spec version changes, run the template validation script to identify fields that need updating." Create a simple validation script (`scripts/validate_templates.py`) that checks each template's frontmatter against the required field list.

### Risk Register Update

The following risks are added to the existing Risk Register (Risks 1-5) in this document.

**Risk 6 — Template tool-call blocks inconsistent with Section 4 spec**

**Probability:** High. The senior-developer will author 17 template files and 10 unit tests in a single iteration. Tool-call blocks in the agent templates (`agent-tool-calling.md`, `agent-pipeline-stage.md`, `agent-mcp-consumer.md`) must strictly conform to the syntax defined in Section 4 of the design spec (fenced `tool-call` blocks with `CALL:`, `purpose:`, `capture:`, `IF:/END IF:` patterns). If the senior-developer invents variations — different key names, different fencing style, different IF syntax — those variations will be propagated into every agent generated from those templates, making the spec non-authoritative.

**Impact:** High. The quality-reviewer and pipeline engine both depend on a single canonical tool-call syntax. Drift in the templates creates a corpus of non-conforming agent files that must be retroactively corrected.

**Mitigation:** The quality-reviewer agent must cross-check every tool-call block in every generated template against Section 4 of the design spec before Milestone 1 is accepted. Specifically:
- Section 4.1 (basic CALL syntax): verify `CALL:`, `purpose:`, `capture:` keys are spelled exactly as specified.
- Section 4.2 (MCP CALL format): verify `mcp__server__tool` naming pattern is used in MCP examples.
- Section 4.3 (CALL with prompt block): verify `prompt: |` multi-line YAML block syntax is correct.
- Section 4.4 (IF/END IF): verify `IF:` and `END IF:` are on their own lines with the condition on the same line as `IF:`.
The work order for the senior-developer (see `docs/spec/senior-dev-work-order.md`) explicitly requires reading Section 4 of the design spec before writing any tool-call blocks and includes a self-review checklist for this purpose.

**Risk 7 — senior-developer.md prompt body not matching the action-plan.md Deliverable 4 content exactly**

**Probability:** Low. The Deliverable 4 content in this action plan is the complete, production-ready file. However, there is a risk that the senior-developer makes "improvement" edits that introduce inconsistencies with the frontmatter or deviate from the Section 4 tool-call syntax.

**Impact:** Medium. The `senior-developer.md` file is the proof-of-concept for the entire `.md` format. If it has errors, it undermines confidence in the approach and requires a second iteration.

**Mitigation:** The work order instructs the senior-developer to write `senior-developer.md` verbatim from Deliverable 4 without creative modification. If the senior-developer identifies a genuine error in Deliverable 4, it must be reported as a scope change rather than silently fixed. The quality-reviewer verifies the file loads via `_load_md_agent()` with the expected field values.

**Risk 8 — Template library at /c/Users/amikinka/.claude/templates/ not accessible from GAIA's CI/CD environment**

**Probability:** Medium. The template library lives outside the GAIA repository at a user-specific local path. CI/CD pipelines run in clean environments without this path. Any test or build step that references the template library will fail in CI.

**Impact:** Low for Milestone 1 (templates are referenced by agents, not by the registry at load time). Medium for Milestone 3 (the ecosystem-builder agent reads templates at runtime; if it runs in CI, it will fail to find them).

**Mitigation:** The template library path must be configurable via an environment variable (e.g., `GAIA_TEMPLATE_LIBRARY_PATH`). The ecosystem-builder agent prompt and the `_load_md_agent()` loader should not hardcode the path. For Milestone 1, this is a documentation note only; for Milestone 3, it is a code requirement. Track as a P2 implementation note in the work order.

**Risk 9 — Milestone 1 scope is too large for a single pipeline iteration**

**Probability:** Medium. Milestone 1 requires: 8 spec edits, 2 registry methods plus a helper, 10 unit tests, 1 agent migration, and 17 template files. For a single senior-developer agent iteration, this is aggressive scope.

**Impact:** Medium. If the senior-developer runs out of iteration budget, some deliverables will be incomplete. The most likely items to be cut are the template library files (Deliverable 2) and the unit tests, since the registry code (Deliverable 3) and the senior-developer.md file (Deliverable 4) are the minimum viable deliverables.

**Mitigation:** The work order specifies a priority sequence. If the senior-developer must cut scope, the following priority order applies:
  - P0 (must complete): spec edits (Deliverable 1) + `_load_md_agent()` + `_build_agent_definition()` (Deliverable 3) + `senior-developer.md` (Deliverable 4)
  - P1 (complete if possible): 10 unit tests
  - P2 (defer to next iteration if needed): template library (Deliverable 2, 17 files)
The software-program-manager will track which items were deferred and prepend them to the next iteration's work order.
