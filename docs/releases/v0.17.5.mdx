# GAIA v0.17.5 Release Notes

GAIA v0.17.5 consolidates custom agents on a single definition format: Python `agent.py`. The legacy YAML manifest format (`agent.yaml` as the sole agent definition) has been removed.

**Why upgrade:**
- **One way to define a custom agent.** The dynamic class builder for YAML manifests is gone â€” every custom agent is now a regular Python class with class-level attributes (`AGENT_ID`, `AGENT_NAME`, â€¦) that you can read, type-check, and refactor with normal tooling. This is the foundation for upcoming external-service work (OAuth-bound integrations such as Gmail and Calendar) where capability declarations live as class attributes the Agent UI can introspect.
- **Smaller, simpler registry.** Roughly 200 lines of dynamic `type()`-based class construction, Pydantic manifest validation, and per-agent MCP-config merging are gone from `src/gaia/agents/registry.py`.

---

## Breaking Change: YAML Manifest Agents Removed

Directories under `~/.gaia/agents/<id>/` that contain only `agent.yaml` (no sibling `agent.py`) are no longer loaded. The registry emits a `DeprecationWarning` per such directory and skips it.

**What still works unchanged:**
- Python `agent.py` agents â€” the canonical way to define a custom agent.
- The optional companion `agent.yaml` sidecar **next to** an `agent.py` whose only purpose is to declare a `models:` preference list.
- The Gaia Builder Agent (`gaia chat --ui` â†’ **+** â†’ **Build a Custom Agent**), which only ever emitted `agent.py`.
- `MCPClientMixin` and per-agent `mcp_servers.json` for Python agents.

**What was removed:**
- `AgentManifest` Pydantic model and the `~80-line _create_manifest_agent_class` dynamic class builder in `src/gaia/agents/registry.py`.
- The `_load_manifest_agent` and `_write_merged_mcp_config` methods.
- The `"custom_manifest"` value of `AgentRegistration.source` â€” the field is now `Literal["builtin", "custom_python"]`.
- `schemas/agent-manifest.schema.json` (the JSON-Schema used by editors for `agent.yaml` autocomplete).

### How to migrate

If you have a `~/.gaia/agents/<id>/agent.yaml` manifest, convert it to `agent.py`. The fastest path is the **Gaia Builder Agent**, which scaffolds a working agent. For a hand-written conversion, see the [Custom Agents guide â†’ Migrating from a YAML Manifest](/guides/custom-agent#migrating-from-a-yaml-manifest), which includes the field-by-field translation table and a worked example.

### Companion-YAML guard

When both `agent.py` and `agent.yaml` exist, the registry reads only `models:` from the YAML. If the sidecar contains any of the legacy manifest fingerprint keys (`manifest_version`, `tools`, `instructions`, `mcp_servers`, `id`), the registry now emits a `DeprecationWarning` listing the ignored keys instead of silently discarding them. Move those fields into `agent.py`.

### Importer behavior for legacy bundles

`gaia` agent bundles exported on v0.17.4 that contain YAML-only directories cannot be imported on v0.17.5. The importer surfaces a clear migration message in `ImportResult.errors` instead of silently dropping the entry â€” re-export the bundle from a v0.17.5 system after migrating each agent to `agent.py`.

### Cleanup of stale files

`_write_merged_mcp_config` previously wrote `~/.gaia/agents/<id>/mcp_servers.json` for manifest agents. After upgrading, those files are inert (no code path reads them for Python agents). They can be safely deleted manually; v0.17.5 does not auto-remove them to avoid touching files a Python agent author may have customized.
