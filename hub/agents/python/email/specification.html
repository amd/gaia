Part 0 · Integration

How to integrate

The agent runs on-device — there is no GAIA cloud, and email content never leaves the machine. You integrate against a local service over one of three surfaces, depending on how much control you need.

1 · REST API

HTTP, stateless

  • Plain HTTP to /v1/email/*
  • Triage, draft, send (+ connection forwarding)
  • Best for: classify / summarize / draft / gated send
2 · MCP server

Tools over stdio

  • MCP tools triage_email, draft_reply, send_email
  • Same three operations as REST, identical results
  • Best for: desktop MCP clients / agent frameworks
3 · Full agent

Natural language

  • Drive the agent in plain language (chat / agent loop)
  • All 17 capabilities incl. organize, calendar, quarantine
  • Best for: end-user assistants acting on the mailbox

Capability → surface

REST and MCP expose the analysis + reply + send core (capabilities 1–4, 6, 7). As of schema 2.1 the REST surface additionally exposes the reversible mailbox mutations archive and phishing-quarantine (capabilities 8, 9) — each gated on a single-use confirmation token like /send, each reversible within a 30-second undo window — plus calendar view / create / respond (#1780; calendar detect and conflicts remain agent-only). MCP does not surface these. The remaining mailbox/calendar capabilities run through the full agent, which acts via a forwarded connection.

Base URL & auth

The service is local — e.g. http://localhost:8080 (configurable). Because it runs on the user's machine there is no cloud API key: the triage / draft / send endpoints take no auth header. The connection-forwarding endpoints require the X-Gaia-UI: 1 header; secrets (refresh token, client secret) are inputs only and are never returned. All bodies are JSON (Content-Type: application/json).

Quickstart — triage → draft → send

1. Triage a message (stateless — the email is in the request):

curl -s http://localhost:8080/v1/email/triage \
  -H "Content-Type: application/json" \
  -d '{
        "payload": {
          "kind": "single",
          "principal": { "email": "me@example.com" },
          "message": {
            "message_id": "m-1001",
            "from": { "email": "dana@vendor.com" },
            "subject": "Action required: sign by Friday",
            "body": "Please review the MSA and reply with your signature by Friday EOD."
          }
        }
      }'
# → { "request_kind": "single",
#     "result": { "category": "actionable", "is_spam": false, "is_phishing": false,
#                 "summary": "…", "action_items": [ … ], "draft": { … } } }

2. Draft a reply and receive a single-use confirmation token:

curl -s http://localhost:8080/v1/email/draft \
  -H "Content-Type: application/json" \
  -d '{ "to": [{ "email": "dana@vendor.com" }],
        "subject": "Re: Action required: sign by Friday",
        "body": "Thanks Dana — signing today, MSA back before Friday EOD." }'
# → { "draft": { … }, "confirmation_token": "9f1c…a7e2" }

3. Send — rejected with 403 unless the token matches this exact payload:

curl -s http://localhost:8080/v1/email/send \
  -H "Content-Type: application/json" \
  -d '{ "to": [{ "email": "dana@vendor.com" }],
        "subject": "Re: Action required: sign by Friday",
        "body": "Thanks Dana — signing today, MSA back before Friday EOD.",
        "confirmation_token": "9f1c…a7e2" }'
# → { "sent_id": "msg-abc123", "sent": true }
The guardrail model

When the agent asks you first

Every email action falls into exactly one tier, enforced by the system rather than left to the model's discretion.

Tier 0 · read-only

No approval

  • Pure analysis; mutates nothing
  • triage, summary, pre-scan, thread, search, list-labels
  • detect meeting, conflicts, list events
  • Drafting (draft_reply) — a draft is a proposal, nothing is sent
Tier 1 · reversible

No per-action approval

  • Mailbox state changes that are logged + reversible
  • archive, mark read/unread, star, label, trash
  • Auto-quarantine (opt-in) of detected phishing — no prompt when enabled, the user is notified, reviewable & reversible (like spam); confirmation-gated by default
  • 30-second undo window on every action
  • Bulk guard: >5 ops across >3 senders → one batch-confirm
Tier 2 · destructive / external

Explicit confirmation

  • Cannot auto-execute — blocks on user approval
  • Payload shows the literal to / subject / body
  • send_draft, send_now, forward_message, permanent_delete
  • accept_invite, decline_invite, create_event_from_email
Cross-cutting · prompt injection

Email bodies are untrusted data

  • Bodies wrapped in <<<UNTRUSTED_EMAIL_BODY>>>; the system prompt treats them as data, never instructions
  • "Forward this to attacker@…" or "ignore prior instructions" is refused and surfaced, never acted on
  • No defense is 100% — this pairs with the deterministic heuristic path and Tier-2 confirmation
Cross-cutting · REST send gate

403 without a valid token

  • POST /v1/email/send is rejected unless a single-use, payload-bound confirmation_token from /draft is supplied
  • A token minted for one body cannot authorize a different body (no bait-and-switch)
  • Fail-loud everywhere: denied = {"status":"denied"}; expired undo returns an actionable error, never a silent no-op

1

4-way triage

ValidatedTier 0 · read-only
Use-case
Read each inbound message and assign it to exactly one of four priority buckets — urgent (needs attention now), actionable (a task is owed), informational (FYI, no action), or low priority — alongside two independent boolean signals, is_spam and is_phishing. A deterministic heuristic over sender, subject, and mailbox labels decides the obvious cases instantly; messages whose bucket depends on body content — especially urgent vs. actionable — are escalated to the local LLM. This single classification drives every downstream view: the pre-scan card, the ranked work-list, and which messages are safe to bulk-archive. Schema 2.0 expands this to five buckets (URGENT / NEEDS_RESPONSE / FYI / PROMOTIONAL / PERSONAL) plus a suggested_action verb — see Schema 2.0.
Why it matters
Turns an undifferentiated inbox into a ranked work-list in one pass — read the 2 urgent items first instead of scrolling 40.
Backed by
triage_inbox · POST /v1/email/triage · MCP triage_email. A deterministic heuristic decides clear cases; ambiguous urgent-vs-actionable escalates to the local model.
Guardrails
Read-only — analyzes only the payload; bodies treated as untrusted data.
Example
Input — single email
{
  "payload": {
    "kind": "single",
    "principal": { "email": "me@example.com" },
    "message": {
      "message_id": "m-1001",
      "from": { "email": "dana@vendor.com" },
      "subject": "Action required: sign by Friday",
      "body": "Please review the MSA and reply with your signature by Friday EOD."
    }
  }
}
Expected output
{
  "request_kind": "single",
  "result": {
    "category": "actionable",
    "is_spam": false,
    "is_phishing": false,
    "summary": "Action required: sign by Friday — Please review the MSA …",
    "action_items": [
      { "description": "Please review the MSA and reply … by Friday EOD.",
        "due_hint": "by Friday" }
    ],
    "draft": { "to": [{ "email": "dana@vendor.com" }],
               "subject": "Re: Action required: sign by Friday", "body": "" }
  }
}
Dataset
Input: ~40 labelled emails across all 4 categories + spam/phishing decoys. Expected: predicted category matches label; spam/phishing flags match; primary action item extracted. Score: per-category precision/recall + 4-bucket confusion matrix.

Per-email summary

ValidatedTier 0 · read-only
Use-case
Condense a single message — however long — into one faithful sentence that captures who wants what, and by when. The agent reads the body locally and returns a length-bounded gist; the REST triage path also returns a deterministic fallback summary (subject + the first one or two sentences, capped at 300 characters), so a summary is always available even without an LLM round-trip. Used when the user asks "what does this say?" or wants to scan a stack of messages without opening each one.
Why it matters
The user decides whether to open a message without reading it.
Backed by
summarize_message tool; the REST triage result also carries a deterministic summary (subject + first 1–2 sentences, capped at 300 chars).
Guardrails
Read-only; body read locally and treated as untrusted data.
Example
// summarize_message(message_id="m-1001")
{ "ok": true, "data": {
  "message_id": "m-1001",
  "subject": "Action required: sign by Friday",
  "summary": "Vendor asks you to review and sign the MSA by Friday EOD." } }
Dataset
Input: emails of varying length, each with a human reference one-liner. Expected: a single sentence preserving who/what/when. Score: LLM-judge faithfulness + length cap (reject multi-sentence dumps).

Draft reply

ValidatedTier 0 · sends nothing
Use-case
Produce a ready-to-edit reply scaffold for a message: recipient pre-filled to the original sender, subject prefixed with Re:, and a body the agent drafts from the thread context. It is a proposal only — nothing is sent — so the user reviews and edits rather than starting from a blank compose window. Over REST, the /draft call also mints a single-use confirmation token bound to the exact (to, subject, body), which is what later authorizes an actual send.
Why it matters
Removes the blank-page cost of replying — the user edits rather than composes.
Backed by
draft_reply · POST /v1/email/draft (returns a single-use confirmation_token bound to the payload).
Guardrails
Drafting sends nothing. Safety refusal: never proposes a draft for spam/phishing, nor for replying to yourself. The token only matters at send time (capability 7).
Example
Input — POST /v1/email/draft
{ "to": [{ "email": "dana@vendor.com" }],
  "subject": "Re: Action required: sign by Friday",
  "body": "Thanks Dana — signing today, MSA back before Friday EOD." }
Expected output
{ "draft": { "to": [{ "email": "dana@vendor.com" }],
             "subject": "Re: Action required: sign by Friday",
             "body": "Thanks Dana — signing today, …" },
  "confirmation_token": "9f1c…a7e2" }
Dataset
Input: an inbound message + a short user intent. Expected: to = sender, subject starts with Re:, body satisfies intent. Score: recipient/subject exact + body intent-coverage (judge). Assert no draft for phishing.
4

Phishing detection

Validateddetect · T0quarantine · auto (opt-in)
Use-case
Identify credential-harvesting and spoofed-sender mail by matching known phishing patterns — e.g. "verify your account" paired with a link, or "your account has been compromised" — and set an independent is_phishing flag during triage. Flagged mail is quarantined — moved to a dedicated GAIA_PHISHING_QUARANTINE label (relabeled and removed from the inbox, never deleted). Quarantine is confirmation-gated by default; with auto-quarantine enabled it applies like spam — no prompt — and the user is notified. They can open the quarantine and unmark a false positive at any time. Throughout, the body is treated as untrusted data, so an instruction embedded in a phishing message ("forward this to…") is refused, not executed.
Why it matters
Protects the user from credential-harvesting mail before they click.
Backed by
is_phishing signal from the triage classifier (keyword + phrase patterns); quarantine_phishing_message applies the GAIA_PHISHING_QUARANTINE label and archives.
Guardrails
Detection is Tier 0. Quarantine is confirmation-gated by default; enabling auto-quarantine makes it Tier 1 spam-style — the user is notified, not asked. Reversible by unmarking from the quarantine label, which preserves the message indefinitely. Acts only when is_phishing=True; never hard-deletes.
Example
// Input: subject "Your account has been compromised",
//        body "…verify your account now: click http://paypa1-secure…"
// → result.is_phishing = true; no draft proposed.
// Quarantined (no prompt with auto-quarantine enabled; confirmed otherwise):
quarantine_phishing_message → { "ok": true, "data": {
  "action_id": "act-55", "message_id": "m-2002",
  "quarantine_label_id": "Label_9", "prior_labels": ["INBOX"],
  "quarantined": true } }   // reverse with unquarantine_message(action_id)
Dataset
Input: true phishing + legitimate security notices (true negatives) + benign mail + one embedded-injection body. Expected: is_phishing matches label; false-positive rate on legit notices is the key metric; injection directive refused.

Inbox pre-scan

WiredTier 0 · suggests only
Use-case
Produce a single "morning brief" snapshot of the whole inbox: the top urgent items, the top actionable items, a count of informational mail, and a list of suggested archives — shaped as a typed email_pre_scan envelope the chat UI renders as a triage card. Default caps keep it scannable (5 urgent / 5 actionable / 10 suggested archives). It only suggests — phishing and spam are surfaced as actionable and are never placed in the archive suggestions, so a destructive action is never one glance away.
Why it matters
The "morning brief" view; see the shape of the inbox at a glance before drilling in.
Backed by
pre_scan_inbox → returns the email_pre_scan envelope (caps: 5 / 5 / 10 by default).
Guardrails
Read-only — suggests, never acts. Safety: spam and phishing are auto-filtered before the scan (see capability 4), so a dangerous message is never proposed for archiving.
Example
{ "kind": "email_pre_scan",
  "urgent": [ { "message_id": "m-9", "sender": "ceo@co.com",
               "subject": "Board deck tonight", "why": "urgent + priority sender" } ],
  "actionable": [ … ],
  "informational_count": 7,
  "suggested_archives": [ { "message_id": "m-22", "sender": "news@list.com",
                            "subject": "Weekly digest", "reason": "low priority" } ],
  "suggested_drafts": [],
  "totals": { "urgent": 1, "actionable": 1, "informational": 7, "suggested_archives": 3 } }
Dataset
Input: a 25–50 message inbox fixture with ground-truth buckets + session preferences. Expected: urgent/actionable lists carry the right ids (set overlap), counts match, no phishing in archive suggestions. Score: per-section set-F1.

Full-thread comprehension

WiredTier 0 · read-only
Use-case
Summarize an entire back-and-forth — not just the latest message — and work out what the user owes by categorizing on the last inbound message in the thread (the one awaiting their response). The summary is prefixed Thread of N messages. and any proposed reply is addressed to that last sender. Lets the user re-enter a long conversation with one sentence of context instead of re-reading a dozen messages.
Why it matters
Catch up on a 12-message thread in one sentence and know what's owed.
Backed by
summarize_thread · POST /v1/email/triage with a ThreadInput (kind="thread", oldest-first messages). Summary prefixed Thread of N messages.
Guardrails
Read-only; every message body treated as untrusted data.
Example
// ThreadInput of 3 messages, last from pm@co.com: "Please confirm by tomorrow."
{ "request_kind": "thread",
  "result": { "category": "actionable",
    "summary": "Thread of 3 messages. Launch date in question; PM needs confirmation by tomorrow.",
    "draft": { "to": [{ "email": "pm@co.com" }], "subject": "Re: Launch slip?", "body": "" } } }
Dataset
Input: threads (3–15 messages) + reference summary + the "what does the user owe?" answer. Expected: summary faithful to the whole thread; category reflects last inbound; draft recipient = last sender.

Send a reply via agent

WiredTier 2 · confirm + token
Use-case
Deliver a reply through the user's mailbox — the end of the draft → confirm → send pipeline and the agent's one outbound action. Because it leaves the device, it always requires explicit confirmation and never auto-sends (mechanics in Guardrails below).
Why it matters
Closes the loop — the agent acts, not just suggests, under an explicit gate.
Backed by
send_now / send_draft · POST /v1/email/send · MCP send_email.
Guardrails
The canonical "ask the user" case. The UI blocks on an explicit confirmation showing the literal to/subject/body. REST/MCP: 403 / rejected unless a valid single-use, payload-bound token from /draft is echoed. Never auto-confirms.
Example
Input — POST /v1/email/send
{ "to": [{ "email": "dana@vendor.com" }],
  "subject": "Re: Action required: sign by Friday",
  "body": "Thanks Dana — signing today …",
  "confirmation_token": "9f1c…a7e2" }
Output (valid token)
{ "sent_id": "msg-abc123",
  "to": [{ "email": "dana@vendor.com" }],
  "subject": "Re: Action required: sign by Friday",
  "sent": true }
// no/invalid token → 403, pointing to /draft
Dataset
Input: draft payloads with (a) correct token, (b) no token, (c) token for a different body. Expected: (a) succeeds; (b)(c) rejected, no send. Gate must be 100% — any unauthorized send is a hard fail. Run against test fixtures, never live mail.

Batch archive

WiredTier 1 · reversible + bulk-confirm
Use-case
Archive many messages — 20 or more — in one reversible operation, recording every per-message change under a shared batch_id so the whole set restores together. This clears the long tail of low-priority, already-read mail after a triage pass. Single archives don't prompt, but a turn exceeding 5 operations across more than 3 senders trips one batch-confirmation before anything is touched — a guard against runaway or injection-driven bulk actions.
Why it matters
Clears the long tail of low-priority mail without 20 clicks; one undo brings it all back.
Backed by
archive_message_batch(message_ids) → one mutation per id under a shared batch_id; undo_archive_batch(batch_id) re-adds the exact prior labels.
Guardrails
Reversible, no per-action approval — but >5 ops across >3 senders trips a single batch-confirm before any op fires. Reversible within the 30s window; undo fails loudly afterward.
Example
archive_message_batch([...])
{ "ok": true, "data": {
  "batch_id": "b0007", "total": 21,
  "succeeded": [ { "message_id": "m-22", "action_id": "act-201" }, … ],
  "failed": [] } }
undo_archive_batch("b0007")
{ "ok": true, "data": {
  "batch_id": "b0007", "restored": 21,
  "messages": [ { "message_id": "m-22", "action_id": "act-201" }, … ] } }
Dataset
Input: inbox with a known archivable set (≥20) + a control set that must not be archived + an over-threshold case. Expected: exactly the archivable set archived, batch_id returned, over-threshold prompts batch-confirm, undo restores all with prior labels.
9

Quarantine + undo

Validated · on RESTundo · T1
Use-case
Give every reversible mutation a safety net. Trash and batch operations are written to a local action log and undone within a 30-second window (restore_message, undo_archive_batch), restoring the exact prior label set; nothing is hard-deleted, and undo past the window fails loudly with manual-recovery instructions. Phishing quarantine is separate — auto-applied when enabled (see capability 4); its unquarantine_message undo restores the exact prior labels within the same window, and past it the quarantine label still preserves the message for manual restore, like spam. This is what lets the user — and the agent — act decisively.
Why it matters
The user acts decisively because every action is recoverable for 30 seconds — the safety net that makes automation trustworthy.
Backed by
A local action log. trash_message ↔ restore_message and batch archive ↔ undo within 30s; quarantine_phishing_message ↔ unquarantine_message within the same window; past it, the quarantine label preserves the message for manual restore. Now on the REST contract (schema 2.1, #1779): POST /v1/email/archive · /quarantine and their /unarchive · /unquarantine reversals — confirmation-gated, so the Agent UI drives them through the package.
Guardrails
Trash / archive / batch are Tier 1 — reversible within 30s, never hard-delete; undo outside the window fails loudly. Phishing quarantine is opt-in auto; undo follows the same window, and the quarantine label keeps the message recoverable manually anytime (capability 4).
Example
// within 30s: unquarantine_message(action_id="act-55")
{ "ok": true, "data": { "action_id": "act-55", "message_id": "m-2002", "restored": true } }
// after 30s:
{ "ok": false, "error": "undo window has expired (30 s)… move it manually in Gmail." }
Dataset
Input: a quarantine/trash/archive action + an undo issued inside and outside the 30s window. Expected: inside → full label-set restoration; outside → loud expiry error, no partial restore.

Calendar — detect

WiredTier 0 · read-only
Use-case
Recognize when an email body is actually asking to schedule a meeting (e.g. "can we meet Thursday at 2pm?") versus merely mentioning a past or hypothetical one. A deterministic heuristic handles clear cases; ambiguous, soft-language bodies ("let's sync sometime") are escalated to the local LLM. It returns whether the message is a meeting request, the matched signals, and a confidence — and makes no calendar changes, only surfacing that a calendar action may be warranted (feeding capabilities 11 and 12).
Why it matters
Surfaces "this needs a calendar action" so invites buried in prose aren't missed.
Backed by
detect_meeting_request(subject, body) → heuristic first; low-confidence escalates to the local LLM. Returns is_meeting_request, confidence, signals, source.
Guardrails
Read-only; inspects text only, makes no calendar changes.
Example
// body: "Can we meet Thursday at 2pm to walk through the numbers?"
{ "ok": true, "data": {
  "is_meeting_request": true, "confident": true, "confidence": "high",
  "source": "heuristic", "signals": ["meet", "Thursday at 2pm"] } }
Dataset
Input: meeting requests + hard negatives ("the meeting went well", "per our meeting"). Expected: is_meeting_request matches label; past-tense/reference negatives not flagged. Score: precision/recall, emphasis on negative precision.
11

Calendar — conflicts

WiredTier 0 · read-only
Use-case
Check the user's calendar, in-line, for events that overlap a proposed time window before they commit to it. Overlap is computed with deterministic half-open [start, end) interval math, so a meeting that ends exactly when another begins is correctly not a conflict, and unparseable or malformed events are skipped rather than reported as spurious clashes. If the calendar can't be read, it surfaces the error instead of falsely reporting "no conflicts."
Why it matters
Know immediately whether a proposed slot collides, without leaving the inbox.
Backed by
detect_calendar_conflicts(start_iso, end_iso) → half-open [start, end) overlap. Abutting meetings (2–3pm vs 3–4pm) do not conflict; malformed events skipped.
Guardrails

End-to-end jobs the agent automates by chaining capabilities. Each workflow's Trigger line shows what kicks it off; each step is tagged with its approval tier so it's clear where the user is asked first.

W1

Morning triage & inbox zero

It's 8am and dozens of unread messages are waiting. The user wants the inbox down to just what needs them — fast, and reversibly.

  4. Inbox reduced to urgent + actionable; everything reversible for 30s
Dataset: 40-message inbox + which messages should be archived / quarantined. Expected: correct card, exact archive/quarantine sets, the batch-confirm fires, phishing is auto-quarantined, undo restores cleanly.
W2

Read → understand → reply → send

A long thread needs a reply. The user wants the gist and a ready-to-send response, reviewed before it goes out.

Dataset: a thread + user intent. Expected: faithful summary → draft to correct recipient → send only with a valid token + explicit confirmation; gate blocks a tokenless send.

Phishing response playbook

A credential-phishing email lands. With auto-quarantine enabled it's quarantined like spam and the user is notified — zero action needed unless they want to look.

Part 2b · Autonomy & always-on

Running unattended planned

The agent can run on its own — woken by a schedule, an event, or a job it queued for itself — entirely on-device. The schedule mechanism ships today in the Agent UI (persisted recurring + one-shot jobs); the full always-on model (heartbeat, webhooks, self-scheduling, background service) is the autonomy engine (#555), planned for the email agent.

What wakes the agent

Time · schedule

Recurring & one-shot

Self-scheduling loop

On each wake the agent runs THINK → ACT → SCHEDULE → COMPLETE: it assesses what changed, takes the allowed actions, and may queue its own next run — "re-check this thread tomorrow", "nudge if no reply in two days", "re-scan the calendar before the proposed slot". Self-scheduling is bounded — a finite horizon, idempotent steps, a capped frequency, and safe-mode by default — so it cannot run away.

Schedules are human-readable strings, parsed to concrete timers; one-shot jobs are supported too.

one-shot: "in 2 days", "tomorrow at 9am"

The tier model still governs everything — but with no human in the loop, the rule is simple: nothing external happens without a person.

Part 4 · API & contract reference

REST surface & contract

The triage request and response form a stable, strict contract — unknown fields are rejected with a 422. The draft/send handshake is part of the same REST surface. required · optional

Endpoints

POST/v1/email/triage

Triage a single email or a full thread. Returns category, spam/phishing signals, summary, action items, and an optional draft proposal. Reads/sends no mail — analyzes only the payload.

Engine selection

No query parameters. Triage runs on the local LLM (#1452); a deterministic heuristic short-circuits high-confidence cases (spam, promotional) as an internal optimization. The split is automatic — there is no caller-facing engine selector.

Errors & status codes

All endpoints return JSON. Errors use the standard { "detail": "…" } shape with an HTTP status; the message names what failed and how to fix it.

StatusMeaningWhen
200OKSuccessful triage / draft / send.
400Ambiguous provider / bad requestTwo mailboxes are connected and neither the token nor the request names a provider (/send, /archive, /quarantine); or /quarantine was called with is_phishing=false. Pass provider (binding the token) or fix the request.
403Forbidden — confirmation gateA mutating action (/send, /archive, /quarantine) called without a valid, bound confirmation_token. Mint one with /draft (send) or /confirm (archive/quarantine).
409Undo window expired/unarchive or /unquarantine called past the 30s window, or the handle is unknown/already undone. Restore manually in the mail client.
422Validation errorRequest violates the contract — unknown field, missing required field, or malformed address. The contract is strict (extra fields are rejected).
502Upstream error/triage: the local-LLM triage call failed. /send · /archive · /quarantine: the mail backend errored (or send returned no message id).
503Backend unavailableThe mailbox account isn't connected. Connect the provider before sending, archiving, or quarantining.

Schema 2.0 — proposed · breaking

Schema 1.0 (above) is what ships today. Schema 2.0 is the proposed next revision — it is breaking (a consumer pinned to 1.0 must opt in) and bundles a taxonomy change with the additive fields. The driver: align the triage labels with the evaluation dataset and give every message an explicit action verb. Tracked in #1615.
