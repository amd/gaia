Email Triage Agent Contract schema_version: 2.3

REST endpoint specification derived from the frozen #1262 contract (gaia_agent_email.contract). Field descriptions are sourced directly from the pydantic models and stay in sync with the contract automatically.

Endpoints

POST/v1/email/triage

Triage a single email or a full thread. Accepts the frozen #1262 EmailTriageRequest and returns a structured EmailTriageResponse — category, spam/phishing signals, a plain-text summary, extracted action items, and an optional reply scaffold. No mail is read or sent; this analyses only the payload in the request. Extracted action items also persist to the local task store, linked to the source message_id and de-duplicated per message (#1605) — the response shape is unchanged.

The draft is a scaffold, not a written reply (schema 2.3): when one is proposed it is a DraftScaffold carrying only to and subject — triage classifies and summarises, it never composes reply prose (so the model choice does not change this). To obtain a sendable reply, compose the body yourself (e.g. an LLM call over the returned summary + action_items + the original message) and pass (to, subject, body) to POST /v1/email/draft, which returns a full DraftReply and a single-use send-confirmation token.

POST/v1/email/triage/batch

Triage a batch of emails or threads in one request (#1887). Accepts a BatchTriageRequest (an items array of 1–100 single-email / thread inputs) and returns a BatchTriageResponse — one BatchItemResult per item, order-preserved. This is additive: the single /v1/email/triage endpoint above is unchanged.

Per-item isolation: a failure on one item sets that entry's error and the rest still run. HTTP 200 with every item errored is valid — inspect each results[].error, not just the status. A 502 means the local LLM was unreachable before any item was processed (the whole batch fails).

POST/v1/email/init

Provision the triage stack and stream terminal-style progress. Tells the running local Lemonade Server to download the configured email model, emitting newline-delimited progress lines (text/plain) a consumer can render line by line. A line beginning ✓ marks success, ✗ a failure; the final line is authoritative.

Scope: the sidecar cannot run the full gaia init or install Lemonade itself. If Lemonade is unreachable this returns 503 with an actionable line and pulls nothing. Once a pull starts the response is a committed 200 (HTTP status cannot change mid-stream), so the trailing ✓/✗ line carries the real outcome. On success, re-run GET /v1/email/init to confirm readiness.

Mailbox actions — archive & quarantine (schema 2.1)

Reversible mailbox mutations exposed on the contract (#1779). Each acts on the mailbox connected in GAIA on the host and is gated on a single-use confirmation token from /v1/email/confirm — the same explicit-confirmation rule as /v1/email/send. Both are reversible within a 30-second undo window via the ungated /unarchive · /unquarantine.

Stateful agent surface

A session-scoped, conversational surface (/v1/email/agent/*) that hosts the full EmailTriageAgent — memory, personalization, and every agent tool — behind an HTTP query interface. Distinct from the stateless triage contract above: this runs the real agent loop and streams it back as Server-Sent Events. This is the surface the Agent UI uses to drive the packaged agent over the network instead of importing it in-process.

POST /v1/email/agent/session

Create (or, with reset:true, recreate) a session-scoped agent. Body: { "session_id": str, "reset"?: bool }. Returns { session_id, created, memory:{ enabled, available, message } }. Building the agent here surfaces construction failures early and warms memory.

POST /v1/email/agent/query

Run one conversational turn and stream the agent loop back as Server-Sent Events (text/event-stream). Body: { "session_id": str, "message": str, "memory_enabled"?: bool }. Each SSE frame is data: {json} with a type of status, thinking, step, tool usage, permission_request (a gated tool is waiting), error, or the terminal run_complete (carrying answer). Because this runs the real agent loop, every agent tool is reachable via natural language. One turn at a time per session — an overlapping call returns HTTP 409.

POST /v1/email/agent/confirm-tool

Approve or deny a tool the agent is blocking on (send / forward / delete / quarantine / calendar-create). Body: { "session_id": str, "approved": bool }. The run resumes when this returns. 404 when no run is awaiting confirmation.

POST /v1/email/agent/cancel

Cooperatively cancel the session's in-flight run. Body: { "session_id": str }.

DELETE /v1/email/agent/session/{session_id}

Evict the session and tear down its agent. 404 if unknown.

GET /v1/email/agent/session/{session_id}/history

Return the conversation so far: { session_id, turns:[{ user, assistant }] } (oldest first).

POST /v1/email/agent/memory

Enable/disable the session agent's memory at runtime (#1666). Body: { "session_id": str, "enabled": bool }. Returns { enabled, available, message }. Enabling memory that was never initialized this session (started with GAIA_MEMORY_DISABLED or Lemonade unreachable) returns HTTP 409 with an actionable message — never a silent no-op.

GET /v1/email/agent/memory/{session_id}

Report the session agent's memory state without changing it: { enabled, available, message }.

Convenience pages

GET /v1/email/spec

This page — a human-readable rendering of the contract above. Not part of the OpenAPI schema.

GET /v1/email/playground

A self-contained, localhost-only playground: a stack-health check plus live triage/draft against this sidecar. Served same-origin with a Content-Security-Policy: connect-src 'self' header, so the page can only reach this sidecar and email content never leaves the machine. Not part of the OpenAPI schema.
